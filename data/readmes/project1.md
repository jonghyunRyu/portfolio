# OJO - CRM Admin Dashboard

## 🔗 배포 URL

AWS: http://high5-ojo.s3-website.ap-northeast-2.amazonaws.com

Vercel: https://frontend-react-virid.vercel.app

## 📝 프로젝트 요약

통신사 고객 데이터를 관리하고 분석하기 위한 **CRM 어드민 대시보드 프론트엔드 프로젝트**입니다.

운영팀이 고객 데이터를 조회하고 분석할 수 있도록  
**고객 관리, 세분화 분석(RFM), 코호트 분석, 지역별 분석 기능을 제공하는 백오피스 시스템**을 구축했습니다.

- 역할: Frontend 개발
- 기술 스택: React, TypeScript, Vite, Tailwind CSS, TanStack Query, Zustand, Chart.js, Mapbox GL

---

# 📖 프로젝트 설명

기존 CRM 운영 환경에서는 고객 데이터를 **여러 화면과 도구에서 분산 관리**하고 있어  
운영팀이 고객 상태를 파악하거나 마케팅 전략을 수립하는 데 많은 시간이 소요되는 문제가 있었습니다.

이를 해결하기 위해 다음 목표로 CRM 어드민 대시보드를 개발했습니다.

- 고객 데이터 **통합 조회 및 관리**
- 고객 행동 기반 **세분화 분석**
- 시각화 기반 **데이터 인사이트 제공**
- 운영팀이 쉽게 사용할 수 있는 **백오피스 UI 구축**
- 고객 관리를 위한 **고객 채널 생성 구현**
---

# 🧩 주요 기능

## 대시보드

운영팀이 핵심 지표를 빠르게 파악할 수 있도록 주요 데이터를 시각화했습니다.

- 고객 수 / 신규 고객 / 이탈 고객 메트릭 카드
- 월별 고객 변화 추이 차트
- 고객 분류 비율 차트
- 상담 카테고리 분포
- 시간대별 상담 현황
- 고객 만족도 카드

---

## 고객 관리

고객 데이터를 효율적으로 조회하고 관리할 수 있는 기능을 제공합니다.

- 고객 목록 테이블
- 다중 컬럼 정렬
- 고객 세그먼트 필터링
- 이름 기반 검색
- 개인정보 마스킹 처리
- 고객 상세 슬라이드 패널

고객 상세 화면에서는 다음 정보를 확인할 수 있습니다.

- 기본 정보
- 고객 특성 정보
- RFM 분석
- LTV 분석

---

## 고객 채널 관리

고객 그룹을 채널 단위로 관리하고, 타겟 마케팅을 수행할 수 있는 기능을 제공합니다.

- 채널 목록 카드 형태 조회
- 채널 생성 (이름, 설명 입력)
- 채널 상세 페이지에서 멤버 목록 확인
- 채널 멤버 추가 / 삭제
- 채널 삭제 (확인 모달 + 완료 알림 후 목록 이동)
- 선택 바구니 패널을 통한 고객 → 채널 연결

### 선택 바구니

고객 목록에서 선택한 고객을 임시 저장하고, 채널에 일괄 추가할 수 있는 기능입니다.

- 고객 목록에서 개별 선택 또는 필터 결과 전체 선택
- 드래그 가능한 플로팅 패널 (채널 목록 + 선택된 고객 2컬럼 구성)
- Zustand + localStorage persist로 페이지 이동 시에도 선택 상태 유지
- 채널 클릭 시 해당 채널에 선택된 고객 일괄 추가
- 새 채널 생성 후 바로 멤버 추가 가능

## 데이터 분석

CRM에서 중요한 고객 분석 기능을 제공합니다.

### RFM 분석

- Recency / Frequency / Monetary 기반 고객 세분화
- 레이더 차트 및 바 차트 시각화
- 세그먼트 기반 마케팅 전략 제안

### 코호트 분석

- 가입 시기별 고객 유지율 분석
- 월별 리텐션 차트

### 지역별 분석

- Mapbox 기반 지도 시각화
- 지역별 고객 분포 확인

# 🎯 문제 상황

## 1. 복잡한 상태 관리 문제

CRM 대시보드는 다음과 같은 다양한 상태가 존재했습니다.

- 고객 데이터
- 필터
- 정렬
- 검색
- UI 상태

초기에는 React state로 관리하려 했지만  
상태 의존성이 많아지면서 **컴포넌트 간 상태 관리가 복잡해지는 문제가 발생**했습니다.

---

## 2. 서버 데이터 관리 문제

대시보드는 다음과 같은 특징을 가지고 있습니다.

- 여러 API 호출
- 동일 데이터 재사용
- 캐싱 필요
- 로딩 상태 관리

단순한 `useEffect + axios` 구조에서는

- 중복 요청
- 캐싱 부재
- 로딩 상태 관리 복잡

문제가 발생했습니다.

---

## 3. UI 구조 확장성 문제

프로젝트 규모가 커지면서

- 기능 단위 코드 분리
- UI 재사용
- 유지보수성

을 고려한 구조가 필요했습니다.

기존의 단순 폴더 구조로는 **기능 단위 관리가 어려웠습니다.**

---

# 💡 해결 방법

## 1. Feature-Sliced Design 아키텍처 도입

프로젝트 구조를 **Feature-Sliced Design(FSD)** 기반으로 재구성했습니다.
```
src
  ├── assets
  ├── pages
  ├── widgets
  ├── features
  ├── entities
  ├── shared
```

효과

- 기능 단위 모듈화
- 코드 의존성 명확화
- 유지보수성 향상

---

## 2. 서버 상태 관리 최적화

서버 데이터 관리를 위해 **TanStack Query**를 도입했습니다.

해결한 문제

- API 캐싱
- 중복 요청 방지
- 자동 refetch
- 로딩 상태 관리

적용 예시

- 고객 목록 조회
- RFM 분석 데이터
- 코호트 분석 데이터

---

## 3. UI 상태 관리 분리

UI 상태는 **Zustand**로 관리했습니다.

예시

- 필터 상태
- 모달 상태
- UI 패널 상태

서버 상태와 UI 상태를 분리하여  
**상태 관리 복잡도를 낮췄습니다.**

---

## 4. 데이터 시각화

분석 데이터를 직관적으로 보여주기 위해 **Chart.js** 기반 차트를 구현했습니다.

사용 차트

- Line Chart
- Radar Chart
- Bar Chart
- Doughnut Chart

---

## 5. 지도 기반 분석

지역별 고객 데이터를 시각화하기 위해 **Mapbox GL**을 활용했습니다.

구현 기능

- 지도 기반 고객 분포 시각화
- 한글 지도 레이블 처리

---

---


## 고객 선택 바구니 상태 관리

고객 선택 바구니는 **Zustand + persist 미들웨어**로 구현했습니다.

- 페이지 이동, 새로고침에도 선택 상태가 유지되도록 localStorage에 자동 저장
- 중복 선택 방지 로직 (Set 기반 ID 비교)
- 전체 선택 시 기존 선택과 병합하는 addMultiple 패턴 적용
- 채널 추가 완료 후 자동 초기화

# 🚀 배포

배포는 브랜치 전략에 따라 두 가지 환경으로 구성했습니다.

- **main 브랜치** → AWS 기반 프로덕션 배포
- **develop 브랜치** → Vercel 기반 협업용 배포

이를 통해 백엔드와 협업 시 빠르게 테스트할 수 있는 개발 환경(develop)과  
실제 배포 환경인 프로덕션(main)을 분리했습니다.

# 인프라 구성

## Production (main)

- 프론트엔드: AWS S3 정적 호스팅
- CI/CD: GitHub Actions

## Development (develop)

- 프론트엔드: Vercel
- 자동 배포: GitHub 연동

# 배포 흐름

## main 브랜치 (AWS)

GitHub Push (main)
→ GitHub Actions 실행
→ npm install
→ npm run build
→ dist 폴더 생성
→ AWS S3 버킷 업로드
→ 정적 사이트 배포

## develop 브랜치 (Vercel)

GitHub Push (develop)
→ Vercel Git Integration 트리거
→ npm install
→ npm run build
→ Vercel Preview Deployment 생성
